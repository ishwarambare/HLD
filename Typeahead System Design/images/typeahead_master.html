💡 Why no Trie? Trie is conceptually useful but impractical at scale — no production DB offers native Trie. With hashmaps: reads are O(1), sharding is native, and we decouple read/write concerns completely.

⚡ Write Optimizations
1
Sampling (Reduce Write Load)
→Only write to DB on every 1000th search
→100K writes/sec → 100 writes/sec (1000x reduction)
→Like exit polls: sample preserves the trend
if (count % 1000 == 0) → write to DB
2
Batching & Threshold Cache
→In-memory cache (1 day) absorbs all writes
→Flush to DB only when count hits threshold
→Prefix updates happen only every 1000 searches — not every search
3
Time Decay — Recency Factor
→Daily: reduce all counts by 1% AND subtract fixed amount
"covid" = 50M
(mostly 2020)
→
"covid" = 49.5M
−1% each day
new_count = (count × 0.99) − 1000 if new_count < 100 → DELETE entry ✅

🔀 Sharding Strategy
→Shard by prefix: "d"→"f" on Shard 1
✅All prefixes of one query → same shard
✅Intra-shard reads = no cross-shard coordination
✅KV store auto-shards by key hash
Shard 1: a→f
Shard 2: g→m
Shard 3: n→t
Shard 4: u→z
🗃️ Caching Layer (Redis)
Cache Hit (<1ms) ~95% of requests
Cache Miss (3-5ms) ~5% of requests
Popular prefixes like "wea", "ipl", "diw" are searched millions of times/min → extremely hot cache with 95%+ hit rate.
🖥️ Client-Side Optimizations
✅Debouncing — wait 200ms after last keypress before firing API. Saves 70-80% of requests.
✅Min 3 chars — don't fire for "d" or "di". Too broad, too noisy.
✅Local cache — cache last few prefix results in browser memory.
Combined: 85–90% fewer API calls from client
