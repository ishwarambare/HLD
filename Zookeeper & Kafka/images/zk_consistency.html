Zookeeper Internals — Leader Consistency & Two-Phase Commit
⚖️ CAP Theorem: Zookeeper is CP — Consistency + Partition Tolerance
📝 Phase 1 — Prepare (Write Request)
1
Write arrives at ZK Leader
A slave DB writes its IP to /master. This always goes to the ZK Leader.
2
Leader writes locally (NON-COMMITTED)
Data is persisted to disk. Status = PENDING. The data exists but is not yet official.
PENDING data on disk ✓
3
Leader replicates to Followers
Leader sends write to all followers. Each follower writes to its own disk (also PENDING).
4
Followers send ACK
Each follower that succeeded sends an acknowledgement. ACK = guarantee: "I can commit this at any time."
ACK: F1 ✓ ACK: F2 ✓ F3 ✗ (partition)
5
Check majority
If ACKs from > half of all machines (n+1 of 2n+1) → proceed to Phase 2. Otherwise → ROLLBACK.
4 of 7 ✓ → PROCEED 2 of 7 ✗ → ROLLBACK

✅ Phase 2 — Commit (or Rollback)
6
Leader commits (majority reached)
Leader changes its own data status from PENDING → COMMITTED.
COMMITTED ✓
7
Leader tells Followers to commit
Sends COMMIT signal to all followers that sent ACK. They update their status to COMMITTED.
F1 COMMITTED ✓ F2 COMMITTED ✓
8
Success response to DB slave
Slave DB that initiated the write now knows it is the new master. ZK notifies all subscribers.
/master = DB-4 ✓
❌ Rollback Path
R
Majority NOT reached → Rollback
If fewer than n+1 ACKs received: Leader discards the PENDING data. Followers told to rollback too. The next competing slave's write gets a chance.
ROLLBACK next slave attempts →
💡 Strong Consistency guarantee: A committed write is guaranteed to be
on at least n+1 machines. Any subsequent read from a majority will
always see this write. No stale reads possible.
